Übersicht über HTML5-Spezifikationen und -Literatur

WHATWG-Spezifikation

  • WHATWG Multi-Page

    Die Spezifikation auf mehrere HTML-Dokumente verteilt

  • WHATWG Single-Page

    Die gesamte Spezifikation in einer riesigen HTML-Datei

  • WHATWG Edition for Web Developers

    Besser lesbare Spezifikation ohne Zusätze speziell für Webentwickler

HTML wird gleichzeitig beim WHATWG-Gremium und bei der W3C-HTML-Arbeitsgruppe entwickelt. Da die HauptautorInnen auf der WHATWG-Seite stehen, ist der erste Anlaufpunkt der täglich aktualisierte Entwurf auf whatwg.org.

Die WHATWG-Spezifikation ist interaktiv: Sie bindet Angaben zur Browserunterstützung ein, ermöglicht Feedback und zeigt offene Fragen (Issues) an. Abgesehen von der Browserunterstützung sind diese Zusätze nur für diejenigen interessant, die die Entwicklung von HTML verfolgen oder zu ihr beitragen wollen.

Die Single-Page-Variante bringt manche Browser zum Einfrieren. Es empfiehlt sich daher die Multi-Page-Spezifikation. Mittlerweile gibt es eine Version der WHATWG-Spezifikation speziell für WebautorInnen. Diese ist ansprechender gestaltet und kommt ohne Zusätze daher.

W3C-Spezifikation

  • W3C Recommendation

    HTML5-Spezifikation beim W3C, stabiler Standard

  • W3C Editor’s Draft

    HTML5-Spezifikation beim W3C, aktueller Autorenentwurf

Beim W3C ausgelagerte Techniken

  • HTML Canvas 2D Context

    JavaScript-API für das zweidimensionale Zeichnen auf dem canvas-Element

    Wichtige Objekte: canvasElement.getContext('2d') uvm.

  • HTML Microdata

    HTML-Attribute, um maschinenlesbare Metadaten unterzubringen, mit JavaScript-API

Die HTML5-Spezifikationen bei der WHATWG und der beim W3C unterscheiden sich: W3C-HTML5 beinhaltet einen Großteil von WHATWG-HTML, lässt allerdings einige Teiltechniken aus oder sie sind in separate Spezifikationen ausgelagert. Zudem gibt es einige inhaltliche Unterschiede.

Der Grund dieser Aufsplittung ist die unterschiedliche Arbeitsweise des Gruppen. Während die WHATWG an einem nicht versionierten »Living Standard« arbeitet, legt die W3C-Arbeitsgruppe einen stabilen, versionierten HTML-Standard festzulegen. Um die Standardisierung zu vereinfachen, klammert sie inhaltlich unabhängige, noch strittige oder unfertige Teile aus.

Einbettbare Auszeichnungssprachen

  • Scalable Vector Graphics (SVG)

    Etabliertes Format für Vektorgrafiken

  • MathML

    Vokabular zur Auszeichnung von mathematischen Formeln

  • Accessible Rich Internet Applications (WAI-ARIA)

    Auszeichnung von Elementen mit »Rollen«, um die Zugänglichkeit zu verbessern

  • RDFa

    Einbettung von RDF-Metadaten in HTML5-Attributen. Konkurrenzformat zu Microdata

Zusätzlich zur Hauptspezifikation gibt es nicht-normative Sekundärliteratur, z.B. Referenzen, Richtlinien und Hintergrundtexte. In diesen Kontext gehören auch weitere Auszeichnungssprachen, die in HTML5 integriert werden können.

Separate JavaScript-APIs (Auswahl)

DOM

  • DOM Living Standard

    Weiterentwicklung und Zusammenführung verschiedener DOM-Spezifikationen. Beinhaltet Core, Events, die Selectors-API (u.a. querySelector) und Element Traversal (u.a. firstElementChild).

Serverkommunikation

  • XMLHttpRequest Living Standard

    Weiterentwicklung von XMLHttpRequest mit cross-domain HTTP-Anfragen, Datei-Uploads und dem Zusammenstellen von Formulardaten.

    Wichtige Objekte: new XMLHttpRequest()

  • Cross-Origin Resource Sharing (CORS)

    Erweitert HTTP um Zugriffsrechte. Darauf baut u.a. cross-domain XMLHttpRequest auf.

    Kernbestandteil: Der HTTP-Header Access-Control-Allow-Origin

  • Server-Sent Events (Teil von WHATWG HTML)

    Eine Server-Push-Lösung mit offen gehaltenen HTTP-Verbindungen. Ermöglicht dem Client, einen kontinuierlichen Datenstrom zu empfangen

    Wichtige Objekte: new EventSource()

  • Web Sockets (Teil von WHATWG HTML)

    Socket-Verbindungen zum Server per JavaScript erzeugen. Nutzt das Web-Sockets-Protokoll

    Wichtige Objekte: new WebSocket()

Datenspeicherung

  • Web Storage (Teil von WHATWG HTML)

    Einfaches clientseitiges Speichern von JavaScript-Daten

    Wichtige Objekte: sessionStorage, localStorage

  • Indexed Database API

    Clientseitige Key-Value-Datenbank mit Indizes

    Wichtige Objekte: indexedDB.open()

  • File API

    Einlesen von Dateien, die NutzerInnen über Formularfelder auswählen hat oder ins Browserfenster ziehen

    Wichtige Objekte: Die files-Eigenschaft bei <input type="file">, new FileReader()

Verschiedenes

  • Web Workers (Teil von WHATWG HTML)

    Hintergrundprozess, der Rechenaufgaben und Datenverarbeitung übernehmen kann. Kommuniziert mit dem Hauptprozess über Nachrichten

    Wichtige Objekte: new Worker(), new SharedWorker()

  • Service Workers

    Spezieller Hintergrundprozess, der für Aufgaben wie Offline-Caching genutzt wird

    Wichtige Objekte: navigator.serviceWorker.register()

  • Geolocation API

    Zugriff auf die geographische Position der NutzerInnen

    Wichtige Objekte: navigator.geolocation.getCurrentPosition()

  • Web Notifications

    Anzeigen von Popup-Meldungen aus Webanwendungen heraus.

    Wichtige Objekte: notifications.createNotification(), notifications.createWebNotification()

  • WebGL

    Zeichnen von dreidimensionalen Objekten auf Canvas mit einem abgespeckten OpenGL-Befehlssatz (OpenGL for Embedded Systems)

„HTML5“ im weiteren Sinne umfasst auch JavaScript-Programmierschnittstellen (APIs), die von verschiedenen Arbeitsgruppen erarbeitet werden.

Die wichtigsten JavaScript-APIs sind bereits Bestandteil der HTML5-Hauptspezifikation, z.B. werden grundlegenden Objekte wie window, document, location und history standardisiert. Dann gibt es APIs, die in der WHATWG-Version integriert und beim W3C ausgelagert sind, so etwa Canvas 2D.

Zusätzliche APIs werden in eigenen Spezifikationen entwickelt. Deren Stabilität und Browserunterstützung variiert stark. Die obige Liste enthält nur die wichtigsten derzeit praxisrelevanten APIs. Die WHATWG hat eine umfangreiche Liste von aktuellen Browsertechniken zusammengestellt.
